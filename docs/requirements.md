# AI 劃詞助手 — 需求文件

> 版本：1.0.4 | 最後更新：2026-06-21

## 1. 專案概述

### 1.1 背景

在瀏覽網頁時，使用者經常需要對頁面上的文字進行翻譯、摘要、解釋或延伸提問。目前的工作流程需要複製文字、切換到 AI 工具分頁、貼上並發送，過程繁瑣且中斷閱讀節奏。此外，不同 AI 供應商的切換也缺乏統一介面。

本專案旨在開發一個 Chrome 擴充套件，讓使用者在任意網頁上劃選文字後，即可就地呼叫 AI 進行對話，實現「所選即所問」的無縼體驗。採用 BYOK（Bring Your Own Key）架構，支援任何 OpenAI 相容的 API 供應商。

### 1.2 目標

- 在任意網頁上劃選文字後，1 秒內出現浮動 AI 圖示可供點擊
- 支援多模態上下文（文字 + 圖片），自動提取選取範圍內的圖片
- 可同時開啟多個獨立 AI 對話框，各自維護對話歷史
- AI 回覆以 SSE 串流即時顯示，支援 Markdown 渲染
- 採用 BYOK 架構，使用者自行設定 API 供應商與金鑰
- 純 Vanilla JS 實作，無框架、無打包工具、無建置步驟

### 1.3 範圍

**In Scope（納入範圍）**
- 劃選文字觸發浮動圖示與對話框
- 多模態上下文提取（文字 + 最多 4 張圖片）
- 多對話框實例管理（拖曳、調整大小、釘選、疊層）
- SSE 串流回應與 Markdown 渲染
- 快速預設問題管理（最多 10 個）
- 逐對話框切換模型
- 右鍵選單整合
- 設定頁面（API URL、Key、模型、快速問題、預設釘選）
- 鍵盤快捷鍵（Enter 發送、ESC 關閉）
- 多語言 UI（55 種語言，Chrome i18n 機制）
- 暗色/亮色模式自動切換（依系統 `prefers-color-scheme`）
- RTL 佈局支援（阿拉伯語、希伯來語、波斯語）
- Base URL 自動補上 `/v1`
- API Key 眼圖示切換顯示
- 快速預設問題行內編輯（單擊編輯、拖曳把手排序）

**Out of Scope（不納入範圍）**
- 非 OpenAI 相容的 API 格式支援
- 本地模型推論
- 對話歷史持久化（頁面重新整理後對話清空）
- 跨裝置對話同步

- **US-007**：作為多供應商使用者，我想要在每個對話框中獨立切換模型，以便針對不同問題使用不同模型

## 3. 具體需求

### 3.1 功能需求

#### FR-001：劃詞觸發浮動圖示
- 使用者在任意網頁選取文字後，游標右下方出現圓形浮動 AI 圖示
- 圖示位置需自動避開視窗邊緣（超出邊界時翻轉至左/上方）
- 若選取範圍僅含圖片（無文字），也應顯示圖示
- 已有未釘選對話框開啟時，不觸發新圖示
- 點擊擴充套件 UI 區域不觸發圖示

#### FR-002：多模態上下文提取
- 選取範圍內的文字以 `Selection.toString()` 提取
- 選取範圍內的 `<img>` 元素以 `Range.cloneContents()` 提取，最多 4 張
- 圖片透過 `fetch` 取得後轉為 DataURL；CORS 失敗時跳過該圖片
- 已是 `data:` URI 的圖片直接使用

#### FR-003：多對話框實例管理
- 可同時開啟多個獨立對話框，每個維護獨立的對話歷史與上下文
- 新對話框以級聯偏移（30px）方式定位，避免完全重疊
- 每個對話框有唯一 ID，點擊時提升 z-index 至最上層

#### FR-004：對話框操作
- **拖曳**：透過標題列拖曳移動，游標限制在視窗範圍內
- **調整大小**：透過 CSS `resize: both` 支援雙向調整
- **釘選**：點擊釘選按鈕切換；釘選時移除遮罩層，點擊外部不關閉
- **疊層管理**：點擊對話框時 `bringToFront()` 提升 z-index
- **關閉**：點擊關閉按鈕或點擊遮罩層（未釘選時）

#### FR-005：SSE 串流回應與 Markdown 渲染
- API 請求使用 `stream: true`
- 透過 `ReadableStream` 讀取回應，逐行解析 `data: ` 前綴
- 每收到 delta 即更新對話框內容，自動捲動到底部
- `data: [DONE]` 結束串流
- Markdown 渲染支援：程式碼區塊（```）、行內程式碼（`）、粗體（**）、斜體（*）、換行
- 所有 Markdown 渲染前先 `escapeHtml()` 防止 XSS

#### FR-006：快速預設問題
- 設定頁可新增、刪除快速問題，最多 10 個
- 對話框中以晶片（chip）形式排列顯示
- 點擊晶片將文字填入輸入框並聚焦

#### FR-007：逐對話框切換模型
- 每個對話框標題列有模型輸入框，附帶 datalist 下拉建議
- 開啟對話框時自動從 API 獲取可用模型列表填入 datalist
- 修改模型輸入框即時更新該對話框的 `config.model`

#### FR-008：右鍵選單整合
- 安裝時建立右鍵選單項目「AI 劃詞助手」，`contexts: ['all']`
- 點擊後透過 `chrome.tabs.sendMessage` 發送 `{ action: 'openDialog' }` 給 content script
- content script 接收後若有選取內容則提取上下文，否則以空上下文開啟對話框

#### FR-009：設定頁面
- **API Base URL**：文字輸入框，placeholder 為 `https://api.example.com/v1`
- **API Key**：密碼輸入框
- **模型名稱**：文字輸入框附 datalist，點擊「獲取」按鈕從 `{baseUrl}/models` 取得列表
- **快速預設問題**：可新增（Enter 或按鈕）、刪除（× 按鈕），上限 10 個
- **預設釘住**：勾選框，開啟後新對話框自動釘選
- **儲存**：驗證必填欄位（API Key、Base URL、模型）後存入 `chrome.storage.sync`
- 所有設定以 `chrome.storage.sync` 同步，鍵值：`apiKey`、`model`、`baseUrl`、`quickPrompts`、`defaultPin`

#### FR-010：鍵盤快捷鍵
- `Enter`：發送訊息
- `Shift + Enter`：輸入換行
- `ESC`：關閉最上層未釘選的對話框

#### FR-011：錯誤處理
- 未設定 API Key 時顯示引導提示
- API 回傳錯誤時在對話框內顯示錯誤訊息（HTTP 狀態碼 + 回應前 200 字）
- 多模態請求回傳 400 時自動重試為純文字模式
- 網路請求例外時顯示錯誤訊息

#### FR-012：多語言支援（i18n）
- 使用 Chrome 內建 `_locales` + `chrome.i18n.getMessage()` 機制
- 支援 55 種語言，`default_locale` 為 `en`
- `manifest.json` 的 `name`/`description` 使用 `__MSG_` 格式
- `popup.html` 透過 `data-i18n*` 屬性標記可翻譯元素，`popup.js` 的 `applyI18n()` 套用翻譯
- `content.js` 透過 `t()` 輔助函式取得翻譯字串
- 系統提示詞為英文，但根據使用者 UI 語言指示 AI 以該語言回應
- RTL 語言（ar、iw、fa）自動設定 `dir="rtl"`，訊息氣泡對齊方向翻轉

#### FR-013：暗色/亮色模式
- 依系統 `prefers-color-scheme` 自動切換，無需手動設定
- `popup.css` 使用 CSS 自訂屬性 + `@media (prefers-color-scheme: dark)` 覆寫
- `content.js` 的 `getThemeColors()` 根據 `matchMedia` 回傳對應色彩組
- 系統切換主題時自動重新注入 content script 樣式
- 滾動條顏色跟隨主題切換

#### FR-014：Base URL 自動正規化
- `normalizeBaseUrl()` 去除結尾斜線
- 若 URL 不以 `/v1`、`/v2` 等版本路徑結尾，自動補上 `/v1`
- 設定頁輸入框失焦時自動修正並儲存
- 所有 API 呼叫（popup 獲取模型、content.js 對話補全）均使用正規化 URL

#### FR-015：API Key 顯示切換
- API Key 輸入框右側有眼圖示按鈕
- 點擊切換 `password` / `text` 類型
- 圖示在張眼/閉眼之間切換

#### FR-016：快速預設問題行內編輯
- 單擊已存在的問題文字即可進入編輯模式
- Enter 或失焦儲存，Escape 取消
- 空字串不儲存（視為取消）
- 僅左側 ⠿ 把手可發起拖曳排序，整列不可拖曳

### 3.2 非功能需求

#### NFR-001：效能
- 浮動圖示需在 `mouseup` 後 10ms 延遲內出現（使用 `setTimeout` 確保選取範圍已確定）
- 串流回應更新時避免不必要的 DOM 重繪，僅更新當前回應的 bubble 元素
- 多對話框拖曳時 `mousemove` 事件需高效遍歷，僅處理 `isDragging` 為 true 的對話框

#### NFR-002：安全
- API Key 僅儲存於 `chrome.storage.sync`，不透過 URL 參數傳輸
- 所有使用者輸入（快速問題、選取文字）在插入 DOM 前必須經 `escapeHtml()` 處理
- 對話框內容渲染前必須先跳脫 HTML 再套用 Markdown 轉換

#### NFR-003：可用性
- UI 文字使用 Chrome i18n 機制，支援 55 種語言，禁止硬編碼 UI 字串
- 暗色/亮色模式自動切換，禁止硬編碼顏色值（使用 CSS variables 或 `getThemeColors()`）
- RTL 語言自動翻轉佈局方向
- 對話框最小尺寸 280×250px，最大不超過 90vw / 90vh
- 圖示與對話框需使用 `all: initial` + `!important` 確保樣式不受宿主頁面影響

#### NFR-004：相容性
- 支援 Chrome Manifest V3
- Content script 需在 `<all_urls>` 正常運作，包含 SPA 與複雜 CSS 的網站
- `content.js` 為 IIFE，載入時不得有副作用（`injectStyles()` 除外）

## 4. 技術要求

- **語言/框架**：純 Vanilla JS（ES2020+），不使用任何前端框架或打包工具
- **平台**：Chrome Extension（Manifest V3）
- **第三方服務**：OpenAI 相容 API（`/v1/chat/completions` + SSE streaming、`/models`）
- **限制與相容性**：
  - Manifest V3 service worker 限制（`background.js` 無 DOM 存取）
  - `chrome.storage.sync` 每項最大 8KB，總計 100KB
  - Content script 需與任意宿主頁面樣式隔離（`__aiext_` 前綴 + `data-aiext` 屬性）
  - 圖片提取受 CORS 限制，失敗時靜默跳過

### 5.3 API 請求範例

**取得模型列表：**
```
GET https://api.example.com/v1/models
Authorization: Bearer sk-xxxx
```
回應：
```json
{
  "data": [
    { "id": "gpt-4o" },
    { "id": "gpt-4o-mini" }
  ]
}
```

**對話補全（串流）：**
```
POST https://api.example.com/v1/chat/completions
Authorization: Bearer sk-xxxx
Content-Type: application/json

{
  "model": "gpt-4o",
  "messages": [
    { "role": "system", "content": "You are an AI assistant. The user selected the following text..." },
    { "role": "user", "content": "解釋這段文字" }
  ],
  "stream": true
}
```
SSE 回應：
```
data: {"choices":[{"delta":{"content":"這段"}}]}
data: {"choices":[{"delta":{"content":"文字"}}]}
data: [DONE]
```

**多模態訊息格式：**
```json
{
  "role": "user",
  "content": [
    { "type": "text", "text": "Here are the images provided by the user as context:" },
    { "type": "image_url", "image_url": { "url": "data:image/png;base64,..." } }
  ]
}
```

## 7. 驗收標準

### AC-001（對應 US-001）：劃詞觸發浮動圖示

**情境一：正常選取文字**
- **Given**：使用者已設定 API Key，且目前無未釘選的對話框開啟
- **When**：使用者在網頁上用滑鼠選取一段文字並釋放滑鼠
- **Then**：選取範圍右下方在 10ms 內出現圓形浮動 AI 圖示

**情境二：靠近視窗邊緣**
- **Given**：使用者選取的文字位於視窗右下角邊緣
- **When**：浮動圖示的預設位置超出視窗邊界
- **Then**：圖示自動翻轉至選取範圍的左方或上方，完整顯示在視窗內

**情境三：已有未釘選對話框**
- **Given**：已有一個或多個未釘選的對話框開啟
- **When**：使用者選取新的文字
- **Then**：不出現新的浮動 AI 圖示

---

### AC-002（對應 US-002）：多模態上下文提取

**情境一：選取範圍同時包含文字與圖片**
- **Given**：使用者選取的範圍內包含文字與 2 張 `<img>` 元素
- **When**：使用者點擊浮動圖示開啟對話框
- **Then**：對話框的上下文預覽區同時顯示選取文字與 2 張圖片縮圖

**情境二：選取範圍圖片超過上限**
- **Given**：使用者選取的範圍內包含 6 張 `<img>` 元素
- **When**：系統提取圖片上下文
- **Then**：僅提取前 4 張圖片，其餘忽略

**情境三：圖片受 CORS 限制**
- **Given**：選取範圍內的圖片因 CORS 策略無法 `fetch`
- **When**：系統嘗試將圖片轉為 DataURL
- **Then**：該圖片被靜默跳過，不中斷流程，其餘可存取的圖片仍正常提取

---

### AC-003（對應 US-003）：多對話框實例管理

**情境一：開啟多個獨立對話框**
- **Given**：已開啟一個對話框 A 並釘選，使用者另選新文字
- **When**：使用者點擊浮動圖示開啟第二個對話框 B
- **Then**：對話框 B 以 30px 偏移開啟，A 與 B 各自維護獨立的對話歷史與上下文

**情境二：對話框疊層切換**
- **Given**：兩個對話框 A（下層）與 B（上層）同時開啟
- **When**：使用者點擊對話框 A 的任意區域
- **Then**：對話框 A 的 z-index 提升至最上層，覆蓋對話框 B

**情境三：關閉其中一個不影響另一個**
- **Given**：兩個對話框 A 與 B 同時開啟
- **When**：使用者關閉對話框 A
- **Then**：對話框 A 的 DOM 與狀態被移除，對話框 B 維持不變

### AC-004（對應 US-004）：拖曳與釘選對話框

**情境三：取消釘選後點擊外部關閉**
- **Given**：對話框已取消釘選
- **When**：使用者點擊對話框外的遮罩層
- **Then**：對話框與遮罩層一起關閉並移除

### AC-007（對應 US-007）：逐對話框切換模型

**情境一：切換模型後發送訊息**
- **Given**：對話框 A 使用模型 `gpt-4o`，對話框 B 使用模型 `gpt-4o-mini`
- **When**：使用者在對話框 A 將模型輸入框改為 `claude-3-sonnet` 並發送訊息
- **Then**：API 請求的 `model` 欄位為 `claude-3-sonnet`，對話框 B 不受影響

**情境二：模型列表自動載入**
- **Given**：對話框開啟時 `baseUrl` 與 `apiKey` 已設定
- **When**：對話框建立完成
- **Then**：自動呼叫 `/models` 取得列表填入 datalist，模型輸入框可下拉選擇

**情境三：模型列表獲取失敗**
- **Given**：API 端點 `/models` 回傳錯誤或無法連線
- **When**：對話框嘗試自動載入模型列表
- **Then**：datalist 為空，使用者仍可手動輸入模型名稱，不顯示錯誤

### AC-010（對應 US-010）：鍵盤快捷鍵

**情境一：Enter 發送訊息**
- **Given**：使用者在對話框輸入框中輸入了文字
- **When**：使用者按下 `Enter` 鍵（未按住 Shift）
- **Then**：訊息被發送，輸入框清空

**情境二：Shift+Enter 換行**
- **Given**：使用者在對話框輸入框中輸入文字

- **Then**：輸入框內插入換行符，不發送訊息

- **Given**：有一個未釘選的對話框開啟
- **When**：使用者按下 `ESC` 鍵
- **Then**：最上層的未釘選對話框被關閉；若所有對話框皆已釘選，則不關閉任何對話框
